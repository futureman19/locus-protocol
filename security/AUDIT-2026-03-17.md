# Locus Protocol Security Audit

**Date:** 2026-03-17
**Auditor:** Claude (Automated Static Analysis + Manual Code Review)
**Scope:** Full codebase — core/ (Elixir), node/ (Elixir), client/ (TypeScript), indexer/ (TypeScript), ghost/runtime/ (Rust)
**Commit:** b044c3c (main)

---

## Executive Summary

The Locus Protocol codebase demonstrates thoughtful security-aware design at the specification level. CLTV lock scripts, capability-based WASM sandboxing, and parameterized SQL queries reflect strong defensive intent. However, the implementation has several gaps between spec and code that introduce exploitable attack vectors.

**Findings by severity:**

| Severity | Count | Description |
|----------|-------|-------------|
| CRITICAL | 1     | WASM sandbox leaks host environment variables to guest code |
| HIGH     | 5     | Replay attacks, missing script paths, float precision in economic formulas |
| MEDIUM   | 7     | Stubbed payment channels, hardcoded values, unrestricted CORS, missing input validation |
| LOW      | 5     | Performance issues, type code mismatches, silent RNG failure, race conditions |

**Key risk areas:**
1. The ghost runtime WASM sandbox inherits the host's full environment, potentially exposing database credentials, API keys, and other secrets to arbitrary WASM code.
2. Ghost invocation nonces are hardcoded to 0, enabling replay attacks on every invocation.
3. The node staking module's CLTV script diverges from the core spec — it lacks the emergency unlock path entirely.
4. Economic calculations use floating-point arithmetic where integer math is required for deterministic consensus.

**Positive observations:**
- Zero SQL injection risk — all indexer queries use parameterized statements
- No hardcoded secrets or credentials in source control
- CLTV lock periods are correctly hardcoded (immutable on-chain)
- Emergency penalty correctly set to 10% (spec-compliant, previous 50% error corrected)
- Capability-based access control design is sound in the ghost runtime

## 2. Cryptographic Review

### 2.1 CLTV Script Construction

**File:** `core/lib/locus/staking.ex:64-85`

The core CLTV lock script uses the correct OP_IF/OP_ELSE/OP_ENDIF structure with:
- Normal path: `<locktime> OP_CHECKLOCKTIMEVERIFY OP_DROP <pubkey> OP_CHECKSIG`
- Emergency path: `<10> OP_CHECKSEQUENCEVERIFY OP_DROP <pubkey> OP_CHECKSIG`

**Issues found:**
- **HIGH:** The `protocol_treasury_address` parameter is accepted but never used in the script (line 64). The emergency path uses the same `owner_pubkey` for both paths — the penalty is enforced at the transaction output level, not the script level. This means the owner could theoretically construct a spending transaction that sends 100% to themselves on the emergency path. See `findings/FINDING-003.md`.
- **HIGH:** `node/lib/locus/staking.ex:35-42` has a completely different, simpler CLTV script with NO emergency unlock path. This divergence means the node and core modules produce incompatible locking scripts. See `findings/FINDING-004.md`.

### 2.2 Transaction Signing

**Files:** `client/src/modules/transaction-builder.ts`, `node/lib/locus/tx_builder.ex`

- Transaction signing is delegated to BSV SDK (`@bsv/sdk`, `bsv_sdk`). No custom signature implementation. This is correct.
- The `signature` field in `buildCityFound` is hardcoded to empty string (`''`) — signatures are not being set. See `findings/FINDING-006.md`.
- Nonce generation in `buildHeartbeat` (line 188) uses `Math.random()` which is **not cryptographically secure**. See `findings/FINDING-007.md`.

### 2.3 Private Key Leakage

**Result: No private key leakage detected.**

- Grep for `privkey`, `private_key`, `secret_key`, `mnemonic`, `seed` across all source files returned zero results in production code paths.
- Private keys are passed as function parameters and never logged or persisted.
- `.env.example` contains only placeholder values; no `.env` file exists in the repository.
- `.gitignore` properly excludes `.env`, `node_modules/`, `.claude/`.

### 2.4 Signature Verification Determinism

- Signature verification uses BSV SDK's OP_CHECKSIG, which is deterministic by definition (secp256k1 ECDSA over SHA256d).
- ID derivation uses `:crypto.hash(:sha256, ...)` — deterministic.
- No custom signature schemes or non-standard crypto detected.

## 3. Economic Attack Vector Analysis

### 3.1 Progressive Tax Formula

**File:** `core/lib/locus/staking.ex:165-167`

```elixir
def territory_tax(base_cost, territory_number) when territory_number >= 1 do
  trunc(base_cost * :math.pow(2, territory_number - 1))
end
```

**Issues:**
- **HIGH:** Uses `:math.pow/2` which returns a float. For `territory_number >= 54`, `2^53` exceeds IEEE 754 double precision integer range. At that point, `trunc()` produces incorrect values. While 54 territories per entity is unlikely, this is a consensus-critical calculation that MUST be deterministic. See `findings/FINDING-005.md`.
- **Client-side:** `client/src/constants/stakes.ts:92-94` has the same float issue with `Math.pow(2, propertyNumber - 1)`.
- No guard against `territory_number = 0` — the `when territory_number >= 1` guard correctly prevents this.

### 3.2 UBI Calculation

**File:** `core/lib/locus/treasury.ex:67-71`

```elixir
def calculate_daily_ubi(treasury_bsv, citizen_count) when citizen_count > 0 do
  daily_pool = trunc(treasury_bsv * @ubi_rate)
  div(daily_pool, citizen_count)
end
```

**Issues:**
- **MEDIUM:** Double truncation — first `trunc()` on the pool, then integer `div()` on per-citizen amount. With `treasury = 10_000_000_000` (100 BSV minimum) and `citizen_count = 21` (minimum for UBI): `daily_pool = trunc(10B * 0.001) = 10_000_000`, `per_citizen = div(10_000_000, 21) = 476_190`. Lost: `10_000_000 - (476_190 * 21) = 10` sats. Negligible at these numbers, but **not an integer overflow risk**.
- **SAFE:** Monthly cap check at lines 101-106 prevents drain. The cap is `trunc(treasury * 0.01)` per month.
- **SAFE:** Minimum treasury of 100 BSV (`@min_treasury_for_ubi = 10_000_000_000`) prevents UBI from activating on empty treasuries.
- No integer overflow risk: Elixir uses arbitrary-precision integers natively.

### 3.3 Fibonacci Block Unlock

**File:** `core/lib/locus/fibonacci.ex:72-78`

```elixir
def blocks_for_citizens(citizen_count) when citizen_count >= 51, do: 24
def blocks_for_citizens(citizen_count) when citizen_count >= 21, do: 16
...
def blocks_for_citizens(_), do: 0
```

**Issues:**
- **SAFE:** No off-by-one errors. Guard clauses use `>=` which matches the spec exactly (51+ = metropolis, 21-50 = city, etc.).
- **SAFE:** Default clause returns 0, not an error. This is correct — 0 citizens means no blocks.
- **NOTE:** `blocks_for_citizens/1` is a pure lookup table, not a Fibonacci calculation. The actual `sequence/1` function (line 40-44) has exponential time complexity but is not called in production paths. See `findings/FINDING-011.md`.

### 3.4 Token Redemption

**File:** `core/lib/locus/treasury.ex:157-195`

```elixir
def redemption_rate(treasury_bsv, total_supply) when total_supply > 0 do
  div(treasury_bsv, total_supply)
end
```

**Issues:**
- **MEDIUM:** `redemption_rate` uses integer division. With `treasury = 100_000_000_000` and `supply = 3_200_000`: rate = 31,250 sats/token. Redeeming 1 token gives 31,250 sats. But `100_000_000_000 / 3_200_000 = 31,250.0` exactly — no loss here.
- **LOW:** Race condition in `redeem_tokens/3` — rate is calculated, then amount is checked against treasury. In a concurrent GenServer scenario, two simultaneous redemptions could both pass the `bsv_amount > city.treasury_bsv` check before either deducts. The GenServer serializes calls, but if the city struct is passed in rather than looked up, stale data could be used. See `findings/FINDING-013.md`.

### 3.5 Minimum Viable Attack Costs

| Attack | Minimum Cost | Mechanism |
|--------|-------------|-----------|
| Found a fake city | 32 BSV (~$1,600 at $50/BSV) | CLTV-locked for 5 months |
| Territory squat (1 territory) | 8 BSV (~$400) | CLTV-locked |
| Sybil (21 fake citizens to enable UBI) | 21 heartbeat transactions (~210 sats in fees) | But requires physical presence proofs |
| Governance attack (51% vote) | Control 51% of citizens + proposal deposit (0.1 BSV) | Requires real identity/presence |
| Treasury drain via UBI | Not possible — 1% monthly cap | Would take 100+ months to drain |
| Emergency unlock to avoid penalty | Not possible — penalty enforced at script level... **except** see FINDING-003 | Script flaw may allow bypass |

## 5. Recommendations Summary

### Immediate (Pre-deployment blockers)
1. Remove `.inherit_env()` from WASM store creation
2. Implement cryptographic nonce generation for invocations
3. Enforce penalty destination in CLTV script (not just at TX output level)
4. Reconcile node and core staking scripts
5. Replace float arithmetic with integer bit-shifting in progressive tax

### Short-term (Before mainnet)
6. Implement network URL whitelist enforcement
7. Replace `Math.random()` with `crypto.getRandomValues()` in heartbeat nonce
8. Add pubkey format validation at all entry points
9. Restrict CORS origins in production
10. Complete payment channel implementation or remove endpoints

### Long-term
11. Add memoization to Fibonacci sequence function
12. Implement proper WASM memory management for input/output
13. Add audit logging for all state-changing operations
14. Implement rate limiting on indexer API
15. Add formal verification for CLTV scripts
